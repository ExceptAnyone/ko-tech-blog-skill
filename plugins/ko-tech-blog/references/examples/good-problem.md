# 좋은 문제 정의 예시

## 예시 1: 성능 문제 + 비즈니스 영향

```markdown
## 문제 정의

사용자가 상품 목록을 볼 때마다 **3초**씩 기다려야 했어요.
1초면 짧지만, 3초는 영원처럼 느껴지죠.

실제로 데이터를 분석해보니 구매 전환율이 **30%나 떨어져** 있었어요.
페이지 이탈률도 **50%**를 넘었고요.

매일 같은 상품 정보를 **100만 번** 이상 데이터베이스에서 조회하고 있었어요.
서버 비용도 매달 증가하는 추세였죠.

> 왜 같은 데이터를 매번 데이터베이스에서 조회해야 할까?
```

**왜 좋은가:**
- 구체적인 숫자(3초, 30%, 50%, 100만 번)
- 사용자 경험에 미친 영향 명확
- 비즈니스 지표(전환율, 이탈률, 비용) 제시
- 블록 인용(>)으로 핵심 질문 강조
- 문제의 근본 원인 암시

---

## 예시 2: 개발 프로세스 문제 + 팀 영향

```markdown
## 문제 상황

배포 한 번 하는 데 **10분**이 걸렸어요.

빌드 → 테스트 → 배포 과정을 모두 거쳐야 했거든요.
작은 오타 수정에도 10분을 기다려야 했죠.

팀원들은 배포를 두려워하기 시작했어요.

"이거 배포해도 될까요?"
"나중에 한꺼번에 배포하죠."

이런 말들이 일상이 되었어요.

하루 배포 횟수가 **2회**에 불과했어요.
빠른 피드백이 필요한 스타트업 환경에서는 치명적이었죠.

> 배포가 이렇게 무거워야 할까?
```

**왜 좋은가:**
- 구체적인 시간(10분)과 횟수(2회)
- 팀 문화에 미친 영향(두려움, 배치 처리)
- 현장 대사 직접 인용
- 비즈니스 맥락(스타트업 환경)
- 블록 인용으로 근본 질문

---

## 예시 3: 코드 복잡도 문제 + 개발자 생산성 영향

```markdown
## 문제 정의

프로젝트에 **15개의 상태 관리 라이브러리**가 섞여 있었어요.

Redux, MobX, Zustand, Recoil, Context API...
각 팀원이 선호하는 라이브러리를 사용한 결과였죠.

새로운 기능을 추가할 때마다 혼란스러웠어요.

"이건 어떤 라이브러리로 관리하지?"
"기존 패턴을 따라야 하나, 새로 만들어야 하나?"

온보딩 시간도 **2주에서 4주**로 늘었어요.
신입 개발자들이 코드베이스를 이해하는 데 너무 오래 걸렸거든요.

버그도 많아졌어요. 상태 관리 방식이 제각각이라
상태 동기화 문제가 자주 발생했죠.

> 왜 우리는 15개의 상태 관리 방법을 유지하고 있을까?
```

**왜 좋은가:**
- 충격적인 숫자(15개)로 시작
- 문제의 원인 명확히 제시
- 개발자 경험에 미친 영향(혼란, 온보딩)
- 질문 형태로 고민 표현
- 2차 영향(버그 증가)도 언급

---

## 예시 4: 데이터 정합성 문제 + 사용자 신뢰 영향

```markdown
## 문제 상황

관리자가 상품 가격을 수정했는데, 사용자에게는 **예전 가격**이 보였어요.

캐시를 도입하면서 생긴 문제였죠.
성능은 개선되었지만, 데이터 정합성이 깨진 거예요.

고객 센터로 문의가 쏟아졌어요.

"결제 페이지 가격이 다른데요?"
"할인이 적용 안 되는데요?"

심각한 건 사용자 신뢰가 떨어진다는 거였어요.
가격이 제각각 보이면, 사이트를 믿을 수 없잖아요.

실제로 장바구니 이탈률이 **20%에서 35%**로 증가했어요.

> 성능과 데이터 정합성, 둘 다 잡을 수는 없을까?
```

**왜 좋은가:**
- 구체적인 문제 사례(가격 불일치)
- 고객 피드백 직접 인용
- 신뢰 문제로 확장
- 정량적 지표(이탈률)
- 딜레마 제시(성능 vs 정합성)

---

## 예시 5: 스케일 문제 + 비용 영향

```markdown
## 문제 정의

서비스 사용자가 **10만 명**을 넘으면서 문제가 생겼어요.

트래픽이 몰리는 저녁 시간대마다 서버가 다운되었거든요.
사용자들은 503 에러 페이지를 봐야 했죠.

서버를 스케일 업하면 해결될까요?
해봤어요. 하지만 비용이 **월 50만 원에서 300만 원**으로 뛰었어요.

근본적인 문제는 **비효율적인 쿼리**였어요.
N+1 문제로 하나의 API 호출에 **수백 개의 쿼리**가 실행되고 있었죠.

모니터링 그래프를 보면 무서웠어요.
피크 타임에는 DB CPU 사용률이 **95%**를 넘었거든요.

> 서버를 늘리는 게 아니라, 코드를 고쳐야 했어요.
```

**왜 좋은가:**
- 스케일 포인트 명확히(10만 명)
- 시도와 실패(스케일 업)
- 비용 영향 구체적 제시
- 근본 원인 분석(비효율적 쿼리)
- 정량적 지표(CPU 95%)
- 블록 인용으로 깨달음 강조

---

## 좋은 문제 정의의 공통점

### 1. 구체적인 숫자
- 응답 시간: 3초
- 전환율 하락: 30%
- 이탈률: 50%
- 비용 증가: 월 100만 원
- 빈도: 100만 번/일

### 2. 영향 범위 명확히
- **사용자**: 느린 경험, 오류
- **비즈니스**: 전환율, 비용, 신뢰도
- **개발자**: 생산성, 온보딩, 버그
- **팀 문화**: 두려움, 배치 처리

### 3. 블록 인용(>)으로 핵심 질문
```markdown
> 왜 같은 데이터를 매번 조회해야 할까?
> 배포가 이렇게 무거워야 할까?
> 성능과 정합성, 둘 다 잡을 수는 없을까?
```

### 4. 현장 대사 직접 인용
```markdown
"이거 배포해도 될까요?"
"사이트가 느려요!"
"결제 페이지 가격이 다른데요?"
```

### 5. 근본 원인 암시
- 반복적인 DB 조회
- 느린 빌드 프로세스
- 비효율적인 쿼리
- 캐시 무효화 누락

## 피해야 할 문제 정의

### 나쁜 예시 1: 추상적 표현
```markdown
## 문제

성능이 좋지 않았습니다.
사용자들이 불편해했습니다.
```

**문제점:**
- "좋지 않았습니다" → 얼마나?
- "불편해했습니다" → 어떻게?
- 숫자 없음
- 영향 없음

---

### 나쁜 예시 2: 기술 나열만
```markdown
## 문제

데이터베이스 쿼리가 최적화되지 않았습니다.
N+1 문제가 있었습니다.
인덱스가 없었습니다.
```

**문제점:**
- 기술적 사실만 나열
- 왜 중요한지 설명 없음
- 사용자/비즈니스 영향 없음
- 공감 요소 없음

---

### 나쁜 예시 3: 원인 없이 결과만
```markdown
## 문제

사용자 이탈률이 높았습니다.
매출이 감소했습니다.
```

**문제점:**
- 왜 이탈했는지 불명확
- 구체적인 숫자 없음
- 근본 원인 없음

---

## 문제 정의 작성 체크리스트

작성 후 다음을 확인하세요:

### 구체성
- [ ] 구체적인 숫자(시간, 비율, 횟수, 비용)가 있는가?
- [ ] 문제 발생 상황이 명확한가?
- [ ] 추상적 표현("좋지 않음", "문제", "불편")을 피했는가?

### 영향
- [ ] 사용자에게 미친 영향을 설명했는가?
- [ ] 비즈니스 지표(전환율, 비용, 이탈률)를 제시했는가?
- [ ] 개발자/팀에 미친 영향이 있다면 언급했는가?

### 공감
- [ ] 현장 대사나 피드백을 직접 인용했는가?
- [ ] 감정이나 상황이 느껴지는가?
- [ ] 독자가 "이건 심각하네"라고 생각할 만한가?

### 구조
- [ ] 블록 인용(>)으로 핵심 질문을 강조했는가?
- [ ] 근본 원인을 암시하거나 명시했는가?
- [ ] 다음 섹션(해결 과정)으로 자연스럽게 연결되는가?

### 톤
- [ ] 기술 나열이 아닌 스토리텔링인가?
- [ ] "왜"가 명확한가?
- [ ] 독자가 문제의 중요성을 이해할 수 있는가?
