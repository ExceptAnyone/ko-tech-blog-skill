# 토스 기술블로그 스타일 가이드

## 핵심 원칙 5가지

### 1. 공감 먼저, 기술은 나중에

**나쁜 예:**
```markdown
# Webpack 번들 최적화 적용

Webpack의 SplitChunksPlugin을 사용하여 번들 사이즈를 최적화했습니다.
```

**좋은 예:**
```markdown
# 배포할 때마다 10분씩 기다리는 건 이제 그만

"또 빌드 실패야?"

프론트엔드 팀원들의 한숨이 깊어졌어요. 배포 한 번 하려면 커피 한 잔 마실 시간이 필요했거든요.
```

**적용 방법:**
- 서론에서 독자가 공감할 만한 일상적 경험이나 문제 상황 제시
- 기술 용어보다는 감정과 상황 묘사로 시작
- "왜"를 먼저 설명하고, "무엇"과 "어떻게"는 나중에

---

### 2. 문제를 먼저 정의하라

**나쁜 예:**
```markdown
# Redis 캐싱 도입

우리는 Redis를 도입했습니다. 다음과 같이 설정했습니다...
```

**좋은 예:**
```markdown
# 같은 데이터를 100번 조회하는 문제

사용자가 페이지를 열 때마다 동일한 상품 정보를 데이터베이스에서 조회했어요.
하루에 100만 번 이상 똑같은 쿼리가 실행되고 있었죠.

> 이 문제를 해결하지 않으면 서버 비용이 매달 2배씩 증가할 상황이었어요.
```

**적용 방법:**
- 해결책 제시 전에 문제를 구체적으로 설명
- 문제의 영향(비즈니스, 사용자, 개발자)을 명확히
- 블록 인용(>)으로 핵심 질문이나 문제의 본질 강조

---

### 3. 단계적 발전 보여주기 (Phase 구조)

**나쁜 예:**
```markdown
# 최종 해결책

최종적으로 다음 구조로 구현했습니다...
```

**좋은 예:**
```markdown
# 해결 과정

## Phase 1: 가장 쉬운 방법부터

"일단 로컬 메모리에 캐싱하면 되지 않을까?"

첫 번째 시도는 간단했어요. Node.js의 Map 객체를 사용해서...

하지만 문제가 있었어요. 서버를 재시작할 때마다 캐시가 날아갔거든요.

## Phase 2: Redis 도입

"영구적인 저장소가 필요해."

Redis를 도입했어요. 캐시가 서버 재시작과 무관하게 유지되었죠...

그런데 새로운 문제가 생겼어요. 캐시 무효화 타이밍을...

## Phase 3: 스마트한 무효화 전략

"언제 캐시를 지워야 할까?"

최종적으로는...
```

**적용 방법:**
- Phase 1/2/3 구조로 시도와 개선 과정 보여주기
- 각 Phase에서 시도 → 결과 → 문제점 → 배움 순서로 작성
- 실패와 시행착오를 솔직하게 공유
- 각 Phase의 제목은 질문형 또는 "무엇을 했는지" 명확히

---

### 4. 정량보다 정성, 하지만 숫자는 강력하다

**나쁜 예:**
```markdown
# 결과

성능이 개선되었습니다.
```

**좋은 예:**
```markdown
# 결과

배포 시간이 10분에서 **30초**로 줄었어요.

더 중요한 건 숫자가 아니었어요. 팀원들이 배포를 두려워하지 않게 된 거죠.
"배포 한 번 할까요?"라는 말이 이제는 가벼워졌어요.

하루 배포 횟수가 2회에서 **15회**로 늘었어요.
```

**적용 방법:**
- 정성적 변화(사용자 경험, 팀 문화)를 먼저 설명
- 핵심 지표는 굵은 텍스트로 강조
- Before → After 숫자 비교
- 숫자에 맥락 부여 (왜 이 숫자가 중요한지)

---

### 5. 기술 용어를 일상 언어로 번역하라

**나쁜 예:**
```markdown
Debounce를 적용하여 API 호출을 최적화했습니다.
```

**좋은 예:**
```markdown
사용자가 검색어를 입력할 때마다 API를 호출하는 건 비효율적이에요.
"토스"를 치는 동안 "ㅌ", "토", "토ㅅ", "토스" 이렇게 4번이나 서버에 요청하거든요.

그래서 **Debounce**를 적용했어요. 쉽게 말하면 "타이핑을 멈춘 후 0.3초 뒤에 검색"하도록 한 거죠.
```

**적용 방법:**
- 기술 용어 사용 전에 일상 언어로 설명
- 비유와 예시 활용
- 기술 용어는 인라인 코드(`)나 굵은 텍스트(**)로 강조
- "쉽게 말하면", "예를 들어", "즉" 같은 연결어 사용

## 문장 패턴

### 서론 패턴

#### 1. 질문형 시작
```markdown
"배포할 때마다 왜 이렇게 오래 걸릴까?"
"사용자들이 느린 이유를 알고 있을까?"
```

#### 2. 공감 유도
```markdown
개발자라면 누구나 한 번쯤 겪어봤을 거예요.
코드를 수정하고, 커밋하고, 푸시하고, 배포하는데...
```

#### 3. 구체적 상황 묘사
```markdown
월요일 아침 9시, 슬랙 알림이 울렸어요.
"사이트가 느려요!"
```

### 문제 정의 패턴

#### 1. 구체적 상황 + 영향
```markdown
사용자가 상품 목록을 볼 때마다 3초씩 기다려야 했어요.
1초면 짧지만, 3초는 영원처럼 느껴지죠.

실제로 구매 전환율이 30%나 떨어졌어요.
```

#### 2. 블록 인용으로 핵심 질문
```markdown
> 왜 같은 데이터를 매번 데이터베이스에서 조회해야 할까?
```

```markdown

```

## 코드 블록 작성법

### 1. Before/After 패턴

```markdown
**Before: 매번 데이터베이스 조회**
\`\`\`typescript
async function getProduct(id: string) {
  // 매번 DB 조회
  return await db.product.findUnique({ where: { id } });
}
\`\`\`

**After: 캐시 먼저 확인**
\`\`\`typescript
async function getProduct(id: string) {
  // 1. 캐시 확인
  const cached = await redis.get(`product:${id}`);
  if (cached) return JSON.parse(cached);

  // 2. 캐시 미스 시 DB 조회
  const product = await db.product.findUnique({ where: { id } });

  // 3. 캐시에 저장 (TTL 1시간)
  await redis.set(`product:${id}`, JSON.stringify(product), 'EX', 3600);

  return product;
}
\`\`\`
```

### 2. 설명 먼저, 코드는 나중에

```markdown
가장 먼저 캐시를 확인해요. 캐시에 데이터가 있으면 바로 반환하죠.

캐시에 없을 때만 데이터베이스를 조회해요. 그리고 조회한 결과를 캐시에 저장하고요.

\`\`\`typescript
// 코드...
\`\`\`
```

### 3. 주석으로 핵심 강조

```typescript
async function optimizedQuery() {
  // ⚠️ 핵심: 캐시 먼저 확인
  const cached = await cache.get(key);
  if (cached) return cached;

  // 💡 포인트: 병렬 처리로 속도 향상
  const [users, products] = await Promise.all([
    fetchUsers(),
    fetchProducts(),
  ]);

  return { users, products };
}
```

## 강조 기법

### 1. 블록 인용 (>)
핵심 질문이나 깨달음 강조

```markdown
> 왜 같은 코드를 여러 곳에서 반복하고 있을까?

> "완벽한 코드보다 빠른 출시가 더 중요할 때가 있어요."
```

### 2. 굵은 텍스트 (**)
중요한 개념이나 키워드

```markdown
우리는 **캐시 무효화 전략**을 세웠어요.
**30초**로 단축되었어요.
```

### 3. 인라인 코드 (`)
기술 용어나 파일명, 명령어

```markdown
`Redis`를 도입했어요.
`package.json` 파일을 수정하고...
`npm run build` 명령어를 실행하면...
```

### 4. 숫자 활용
구체성과 신뢰성 부여

```markdown
3가지 방법
10배 개선
0.3초 응답
15회 배포
```

## 문단 리듬

### 짧은 문단 (1-3줄)
- 강조하고 싶은 포인트
- 질문
- 전환점

```markdown
하지만 문제가 있었어요.

서버를 재시작하면 캐시가 모두 사라졌거든요.

"영구적인 저장소가 필요해."
```

### 긴 문단 (5-7줄)
- 상황 설명
- 과정 묘사
- 배경 설명

```markdown
첫 번째 시도는 간단했어요. Node.js의 Map 객체를 사용해서 메모리에 데이터를 캐싱했죠.
구현도 쉬웠고, 성능도 좋았어요. 응답 시간이 3초에서 0.5초로 줄었거든요.
하지만 곧 문제를 발견했어요. 서버를 재시작할 때마다 캐시가 날아가는 거예요.
배포할 때마다 사용자들이 느린 페이지를 경험해야 했죠.
```

### 교대 배치
```markdown
[긴 문단: 상황 설명]

[짧은 문단: 강조]

[긴 문단: 과정 묘사]

[짧은 문단: 질문]
```
